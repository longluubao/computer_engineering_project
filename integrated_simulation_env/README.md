# Integrated Simulation Environment (ISE) for AUTOSAR SecOC + PQC

> **Purpose.** A **PQC-and-SecOC-protocol harness** that links the real
> Post-Quantum Cryptography modules from `Autosar_SecOC/source/PQC/` and
> reproduces the AUTOSAR SecOC frame layout (header + 64-bit freshness +
> authenticator) in a deterministic single-process driver. It produces the
> **performance and security evidence** needed for the thesis under
> realistic automotive scenarios (multi-ECU, multi-bus, attack injection).
>
> **What the ISE actually links** (verifiable in `CMakeLists.txt`):
>
> - `Autosar_SecOC/source/PQC/PQC.c` — ML-KEM-768 + ML-DSA-65 wrapper
> - `Autosar_SecOC/source/PQC/PQC_KeyDerivation.c` — HKDF session-key derivation
> - `Autosar_SecOC/source/PQC/PQC_KeyExchange.c` — ML-KEM handshake
> - `external/liboqs/build/lib/liboqs.a` — NIST FIPS 203/204 implementation
>
> **What the ISE does NOT link, and where its conformance evidence comes from
> instead:** the upper AUTOSAR Basic Software stack (Com, PduR, SecOC.c,
> Csm, CryIf, CanIf, CanTp, SoAd, NvM, Det, Dem, BswM, Os, …) is **not**
> compiled into `ise_runner`. Conformance for those modules is asserted by
> the **40-file gtest unit-test suite under `Autosar_SecOC/test/`** which
> links the real `SecOCLib` static library (8 950 LOC of test code, 678
> individual test cases, 41/41 ctest executables passing — see
> `Autosar_SecOC/test/`). The ISE measures the PQC pipeline + the SecOC
> protocol invariants (freshness monotonicity, attack detection,
> deadline tracking) at integration scale; the unit tests measure
> AUTOSAR API conformance per module. Together they form the
> two-layer evidence base of the thesis.

## 1. What you get

1. **Virtual ECU network** — 3 ECUs (Tx, Rx, Gateway) running the real BSW
   stack in three threads / processes.
2. **Virtual buses** — CAN 2.0 B, CAN-FD, FlexRay TDMA, and 100BASE-T1 /
   1000BASE-T1 Ethernet emulated with configurable bandwidth, MTU, bit error
   rate (BER) and propagation delay.
3. **Signal catalogue** — 14 representative automotive signals classified by
   ASIL (QM…ASIL-D), cycle time (5 ms…1 s) and bus (see
   `docs/SIGNAL_CATALOGUE.md`).
4. **Attack library** — 10 parameterised attackers: replay, bit-flip tamper,
   key-confusion MITM, freshness rollback, DoS flood, signature fuzz,
   downgrade-to-HMAC, harvest-now-decrypt-later, side-channel timing probe,
   quantum-oracle simulation.
5. **Metric collector** — latency histograms (p50/p95/p99/p99.9/max), jitter,
   throughput (msg/s & MB/s), overhead (Δ vs. unauthenticated), deadline-miss
   rate, attack-detection rate, false-positive rate.
6. **Thesis-ready reports** — CSV raw + JSON summary + Markdown tables + PNG
   plots generated by `reports/generate_thesis_report.py`.

## 3. How to build & run

### Prerequisites

Same as the main project — OpenSSL, liboqs, CMake ≥ 3.20, GCC/Clang, Python
≥ 3.9 (only for reports).  Windows users can use MinGW; on Linux any recent
GCC works.

### One-shot build

```bash
cd integrated_simulation_env
bash build.sh         # configures CMake and builds ise_runner
```

### Run the full thesis suite (all scenarios, 3 repetitions)

```bash
bash run_all_scenarios.sh
# → writes results/<UTC-timestamp>/{raw/,summary/,plots/}
```

### Run a single scenario manually

```bash
./build/ise_runner --scenario baseline_classic \
                   --bus can_fd --iterations 1000 \
                   --out results/quicktest
```

### Generate the thesis report

```bash
python3 reports/generate_thesis_report.py \
    --input results/<timestamp> \
    --output results/<timestamp>/report.md
```

## 4. What goes into the thesis

| Thesis chapter                | Artifact produced by ISE                                       |
|-------------------------------|----------------------------------------------------------------|
| Performance analysis          | `results/<t>/summary/latency_stats.csv`, plots                 |
| Bus / bandwidth discussion    | `results/<t>/summary/bus_utilisation.csv`                      |
| Security evaluation           | `results/<t>/summary/attack_detection.csv`                     |
| Real-time compliance          | `results/<t>/summary/deadline_miss.csv`                        |
| AUTOSAR conformance           | `results/<t>/summary/api_coverage.json`                        |
| PQC vs classical comparison   | `results/<t>/summary/pqc_vs_hmac.csv` + plots                  |
| ISO 21434 / UN R155 evidence  | `results/<t>/summary/compliance_matrix.md`                     |

All CSV files carry enough columns (mean, min, max, stddev, p50, p95, p99,
sample count, seed, timestamp) to be dropped straight into the thesis with
`pandas`, Excel or LaTeX `pgfplotstable`.

## 5. Relation to previous work

- Unit tests in `Autosar_SecOC/test/*.cpp` — kept as-is, executed by CTest.
- Phase-1..4 tests in `PiTest/` — kept as baseline reference.
- The ISE is **strictly additive** — nothing in the main project is
  modified. If `SecOCLib` is built, the ISE links it; otherwise it builds
  a thin stand-alone variant from the same sources.

See `ARCHITECTURE.md` for how the simulator binds to the stack, and
`THESIS_RESEARCH.md` for the automotive and cryptographic constraints that
drive the scenario design.
